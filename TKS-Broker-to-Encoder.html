Technical Knowledge Share: Broker to Encoder Pipeline

Uplynk Ingest Pipeline — Transcoding Workflow Deep Dive

Overview

This TKS covers the Broker-to-Encoder segment of the Uplynk ingest pipeline — the core transcoding workflow that takes source video slices and produces multi-bitrate segments for adaptive playback.

The Slicer encodes only the top bitrate. Everything below the top bitrate is the Broker and Encoder's job.

Context: Where Broker → Encoder Fits

flowchart LR
    SV[Source Video] --> SL[Slicer]
    SL -->|Top bitrate slice| BR[Broker]
    BR -->|Distribute slices| EN[Encoder]
    EN -->|Transcoded slices| S3[Storage - S3]
    EN -->|Metadata| MDB[(MongoDB)]
    EN -->|Success/Failure| BR
    BR -->|ACK| SL
Important: This is NOT the only ingest path. Dasher bypasses Broker and Encoder entirely, uploading directly to S3 (one Dasher instance per bitrate). This TKS focuses exclusively on the Broker → Encoder path.

Component Responsibilities

Broker Responsibilities

ResponsibilityDetail
Receive slicesAccepts top-bitrate slices from one or more Slicers
Distribute workFans out slices to available Encoders for lower-bitrate transcoding
Track completionMonitors Encoder success/failure per slice
Retry on failureIf an Encoder fails or doesn't respond, retries with a different Encoder
Notify SlicerOn successful encode of all bitrates, ACKs back to Slicer

Encoder Responsibilities

ResponsibilityDetail
Receive slicesAccepts slices from exactly one Broker via WebSocket
TranscodeEncodes slice into target lower bitrates (.ts / .m4s) per the Encoding Profile
Upload to S3Stores transcoded slices in the nearest regional S3 bucket
Report backSignals success or failure to the Broker
Retry storageOn S3 upload failure, retries in other regions before reporting failure

Communication Flow

sequenceDiagram
    participant SL as Slicer
    participant BR as Broker
    participant EN as Encoder
    participant S3 as S3 Storage
    participant DB as MongoDB

    SL->>BR: Upload top-bitrate slice
    BR->>BR: Queue slice for transcoding
    BR->>EN: Distribute slice (WebSocket)
    EN->>EN: Transcode to lower bitrates
    EN->>S3: Upload transcoded slices
    S3-->>EN: Upload ACK
    EN->>DB: Write slice metadata (beams collection)
    EN-->>BR: Report success
    BR-->>SL: ACK slice complete

  • Many Slicers → One Broker: Multiple slicers connect to a single broker
  • One Broker → Many Encoders: Broker distributes across a pool of encoders via WebSocket
  • Each Encoder connects to exactly one Broker at a time
  • Encoders are scaled by the encoder autoscaler type per zone

Diagnosing Issues: Slicer-Side vs Platform-Side

flowchart TD
    A[Elevated broker_debugCreationLiveMax?] -->|Yes| B{broker_maxage_live also elevated?}
    B -->|Yes| C[Platform-side issue]
    B -->|No| D[Slicer-side issue]
    C --> C1[Broker/Encoder backlog]
    C --> C2[Storage write failure]
    C --> C3[Encoder impairment]
    D --> D1[Slow slicer encode]
    D --> D2[Network issue uploading to Broker]
    D --> D3[Slicer host resource exhaustion]
Key insight: broker_maxage_live is platform-side only (Broker receives → Encoder completes). broker_debugCreationLiveMax is end-to-end including slicer time. If only debugCreationLiveMax is elevated but maxage_live is normal → the delay is on the slicer side.

Autoscaling

Both Brokers and Encoders are managed by the Uplynk autoscaler system.

Autoscaler TypeWhat It ScalesPipeline

slicerCloud Slicer workersIngest

Impairment PromQL Queries

# Currently impaired encoders by zone
sum by (ulscalerzone) (autoscaler_scaler_impaired_servers2{ulscalertype="encoder"})

# New encoder impairment events
sum by (ulscalerzone) (increase(autoscaler_scaler_impaired_server_stopping_total{ulscalertype="encoder"}[15m]))

Storage Write Path

Encoders write transcoded slices to S3 with this logic:

  1. Write to nearest regional S3 bucket (same region as the encoder)
  2. On failure → retry in a different S3 region
  3. On all-region failure → report failure back to Broker
  4. Each slice is stored in exactly one region (no replication)

Available S3 regions: us-east-1, us-east-2, us-west-2, eu-central-1, ap-south-1, ap-southeast-1, sa-east-1

Common Failure Patterns

SymptomLikely CauseInvestigation
broker_maxage_live spikes across all zones Encoder capacity issue or S3 regional outage Check broker_enc_busy vs broker_encoders, check impairment metrics
broker_maxage_live spike in single zone Zone-specific encoder impairment Check autoscaler_scaler_impaired_servers2{ulscalertype="encoder", ulscalerzone="..."}
WebSocket drops increasing Network instability between Broker and Encoder instances Check zone health, recent deploys
broker_transcode_errors_total rising Encoder software bug or bad input slices Check encoder logs, correlate with recent deploys
broker_upload_failures_total rising S3 availability issue Check CloudWatch S3 metrics, check if retries to other regions succeed
broker_beam_ready_total dropping Slicers disconnecting from Broker Slicer-side or Services API issue (broker discovery)

Key Takeaways

  1. Broker is the orchestrator — it doesn't transcode, it distributes and tracks
  2. Encoder is the workhorse — transcode + upload + report back
  3. WebSocket is the connection between Broker and Encoder (fragile to network issues)
  4. Retry logic exists at every hop — Broker retries encoders, Encoders retry S3 regions
  5. broker_maxage_live is your primary signal for platform-side ingest health
  6. Not all ingest goes through this path — Dasher is the exception
